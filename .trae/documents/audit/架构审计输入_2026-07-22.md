# fineSTEM 架构审计输入

**用途**：交由架构 agent 对 fineSTEM × ZeroClaw 集成做彻查。
**背景**：刚完成一次大规模重构（ZeroClaw 集成重构 + 测试体系重建），148/148 测试通过。但重构中发现若干架构层面的债和待定问题，需要架构层面定夺，不是代码 patch 能解决的。

---

## 一、项目是什么

fineSTEM 是面向 12-18 岁青少年的 STEM 项目式学习（PBL）平台。学生通过 AI 导师引导，走完 9 阶段流程（脑爆→开题→范围→轨道→设计→计划→编码→验收），产出**可运行成品 + 研学文档**。

**不是**豆包编程（AI 替用户生成代码），**是** AI 研学导师（教学生做项目）。

## 二、技术栈与运行时

| 层 | 技术 | 说明 |
|----|------|------|
| 前端 | React 18 + Vite + TypeScript + Tailwind | SPA，3312 行的 Create.tsx 是核心页面 |
| 后端业务 | FastAPI + Python | CRUD、鉴权、工件存储、代码沙箱编排 |
| AI 运行时 | **ZeroClaw**（Rust 编译的 daemon，0.8.3） | 独立进程，监听 127.0.0.1:42617 |
| AI 模型 | DeepSeek（主）+ GLM（回退） | 通过 ZeroClaw 的 provider 抽象 |
| 工具协议 | MCP（stdio JSON-RPC 2.0） | fineSTEM 后端 12 个 PBL 工具通过 MCP server 暴露给 ZeroClaw |
| 数据库 | sqlite | `D:/data/finestem/finestem.db`（Windows D 盘绑死）|

### 关键事实：AI 主链路绕过了 FastAPI

前端 `useStreamingChat.ts` **直连 ZeroClaw WebSocket**（`ws://127.0.0.1:42617/ws/chat`），**不经过 FastAPI 的 orchestrator**。FastAPI 只负责 CRUD（项目/Demo/成果卡/鉴权）。ZeroClaw 通过 MCP 回调 FastAPI 后端的工具。

```
前端 ──ws──> ZeroClaw daemon ──MCP stdio──> fineSTEM 后端 tools.py（12 个工具）
                                    │
                                    └──HTTP──> DeepSeek API
```

## 四、需要架构 agent 彻查的问题（按重要性排序）

### 问题 A：双系统并存——orchestrator.py（3348 行）是死代码还是备份？

**现状**：
- `orchestrator.py`（3348 行）实现了完整的 AI 编排逻辑：system_prompt 构建、skill 路由、tool calling loop、截断续接、阶段门禁、question 解析。
- 但前端**不走 orchestrator**——直连 ZeroClaw。`api/agent.py` 的 WebSocket 端点 `/api/v1/agent/ws` 没有前端调用方。
- orchestrator 自己还有 11 处 `STAGE_ORDER` 定义（没迁到新的 stage_constants.py），和 stage_constants.py 形成第二套门禁常量。
- orchestrator 内部的门禁检查（如 `_execute_tool` 对 artifact_writer 的 path 检查）**是死代码**——取 `params["path"]` 但工具真实字段是 `artifact_name`，永远不命中。

**疑问**：
1. orchestrator 是否应该彻底删除？还是保留作为"ZeroClaw 不可用时的 FastAPI 回退路径"？
2. 如果保留，两套门禁常量（orchestrator 自有的 vs stage_constants.py）如何统一？现在它们会漂移。
3. `skill_loader.py` / `skill_registry.py` 只被 orchestrator 调用——orchestrator 死了它们也死了。原 `.trae/skills/stem-pbl-guide/`（给其他 IDE）还要不要这个加载器？

### 问题 B：ZeroClaw 配置不在版本控制

**现状**：
- `H:/dev-env/zeroclaw/config/config.toml`、`workspace/` 下的 SOUL.md/IDENTITY.md、新建的 skill SKILL.md——全部在仓库外（`H:/dev-env/`），不在 git。
- 本次 config.toml 字符串闭合 bug 就是因为没有版本控制和 review 流程，改坏了 1 个月没人发现。
- 备份是手工 `.bak` 文件，不可追溯。

**疑问**：
1. ZeroClaw 配置是否应该纳入 fineSTEM 仓库（如 `deploy/zeroclaw/`）？还是作为"部署环境配置"保持在仓库外？
2. 如果纳入，如何处理路径差异（开发机 `H:/dev-env/` vs 其他环境）？
3. config.toml 里的 system_prompt（250 行）和 workspace 下的 SKILL.md 内容重叠/分裂——哪个是 source of truth？

### 问题 C：双份 skill 维护

**现状**：
- `.trae/skills/stem-pbl-guide/SKILL.md`（2621 行）——给 Trae/Cursor 等 IDE 用，内容完整但**部分过时**（还在用废弃的 `<question>` XML 格式，而 ZeroClaw 已改用 `ask_question` 工具）。
- `H:/dev-env/zeroclaw/config/agents/assistant/workspace/skills/stem-pbl-guide/SKILL.md`（230 行）——ZeroClaw 专用，内容新（用 ask_question 工具），但不完整（没有原版的 schema/templates/research-docs）。
- 两份独立维护，内容会漂移（已经漂移了）。

**疑问**：
1. 是否应该合并成一份 SKILL.md，通过变量/条件适配不同运行时？
2. 还是接受双份，但建立同步机制（如 lint 检查关键规则一致性）？
3. ZeroClaw 的 skill 机制是否支持 `references/` 子目录引用外部文件？如果能，能否让 ZeroClaw skill 引用原版的 schema/templates，避免重复？

### 问题 D：Create.tsx 3312 行——需要拆分吗？

**现状**：
- `Create.tsx` 是单文件 3312 行，包含：聊天 UI、PBL 流程状态机、代码编辑器联动、教学模式、question 卡片管理、消息清洗（sanitizeAssistantNarration）、WebSocket 事件处理、项目创建、代码提取……
- 状态管理全在组件内（useState/useRef 共 40+ 个），没有用 Context/Redux/Zustand。
- `sanitizeAssistantNarration`（150 行清洗逻辑）虽然已简化，但仍内联在 Create.tsx 里，测试时需要 export 出来才能测。
- 本次重构改 Create.tsx 时，因为它太大，多次编辑出现重复代码块（已修复，但说明可维护性差）。

**疑问**：
1. Create.tsx 是否应该拆分？如果拆，边界在哪？（候选：chat 逻辑、PBL 状态机、清洗函数、代码编辑器联动）
2. 状态管理是否应该抽到独立 store？40+ useState/useRef 在单组件里，竞态风险高（已经遇到 isLoadingRef 的竞态问题）。
3. 拆分的优先级如何？（功能正确性已验证，但可维护性债在累积）

### 问题 E：ZeroClaw 的 SOP 机制是否应该用于 PBL 阶段门禁？

**现状**：
- PBL 门禁目前用两层实现：① config.toml system_prompt 里告诉 AI 规则 ② 后端 tools.py 的 stage_advancer/artifact_writer 做硬门禁。
- 但门禁是"软"的——AI 可以选择不调 stage_advancer（虽然 system_prompt 要求它调），系统无法强制 AI 按顺序走。
- ZeroClaw 支持 SOP（Standard Operating Procedure）——声明式节点图 + 审批流，AI 必须按图走。

**疑问**：
1. 用 SOP 把 9 阶段 PBL 定义成节点图，是否比"prompt 规则 + 工具硬门禁"更可靠？
2. SOP 的学习成本和维护成本如何？是否值得？
3. 还是当前"prompt + 工具门禁"的组合已经够用（148 测试通过，实测 4 轮 PBL 对话正常）？

### 问题 F：测试目录污染（58 个文件，18 个调试脚本）

**现状**：
- `apps/backend/tests/` 有 58 个 `.py` 文件，其中约 18 个是一次性调试脚本（`test_simple.py`、`test_debug.py`、`test_source.py` 等），硬编码 `G:\` 路径，靠模块级副作用跑，无断言结构。
- 2 个文件（`test_pbl_dialogue_flow.py`、`test_stream_truncation.py`）依赖已废弃的 orchestrator 符号，用 `pytestmark = skip` 停用。
- `conftest.py` 硬编码 `DATABASE_URL=sqlite:///D:/data/finestem/test_finestem.db`（Windows D 盘绑死）。

**疑问**：
1. 18 个调试脚本应该清理还是归档？
2. 测试数据库路径如何解耦（开发机 vs CI）？
3. 是否需要建立测试准入标准（新测试必须符合什么规范才能进 tests/）？

### 问题 G：Windows 路径/编码硬编码散落

**现状**：
- `conftest.py`：`DATABASE_URL=sqlite:///D:/data/finestem/...`
- MCP server 的 config.toml：`FINESTEM_DB_URL = "sqlite:///D:/data/finestem/finestem.db"`
- `_trigger_auto_export` 的 `out_dir` 候选路径逻辑复杂，依赖 cwd
- Python 脚本在 Windows GBK 控制台崩溃（已修但说明编码假设散落）

**疑问**：
1. 这些硬编码是否应该集中到 `.env` / settings？
2. 是否应该支持非 Windows 部署（目前多处假设 Windows 路径分隔符和 D 盘）？
