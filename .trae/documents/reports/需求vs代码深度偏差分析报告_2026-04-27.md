# fineSTEM 需求 vs 代码：深度偏差分析报告

**版本**: v1.0
**日期**: 2026-04-27
**状态**: 审计报告
**对应需求文档**: `04_fineSTEM_BS平台产品方案_V3.3.md`
**审计范围**: 前端 `apps/frontend/` + 后端 `apps/backend/` 全量代码 vs V3.3 产品方案 + UI 布局规格 + MVP 成功判据

---

## 0. 总体判断

**代码实现与 V3.3 产品方案存在严重偏离。** 核心问题不是"功能没做完"，而是**开发方向跑偏了**——大量精力投入到了需求文档中根本不存在的功能模块（升学、问卷引擎、知识来源），而需求中定义的核心交互流程和 UI 布局却未按规格落地。

---

## 1. 升学菜单：需求中不存在，代码中完整实现

### 1.1 需求定义

V3.3 产品方案全文 **没有任何升学相关功能**。产品定位是"青少年 STEM 研学助手平台"，核心闭环是 `Demo → Fork → 轻项目 → 成果档案卡 → 分享`。

### 1.2 实际代码

升学功能在前端和后端都有**完整实现**：

| 层级 | 文件 | 功能 |
|------|------|------|
| 前端页面 | `apps/frontend/src/pages/HongKongMacaoAdmissions.tsx` | 港澳升学计划 CRUD |
| 前端页面 | `apps/frontend/src/pages/InternationalAdmissions.tsx` | 国际升学计划 CRUD |
| 前端页面 | `apps/frontend/src/pages/ProfileEnhancement.tsx` | 背景提升计划 CRUD |
| 后端 API | `apps/backend/app/api/hongkong_macao.py` | 港澳升学 REST API |
| 后端 API | `apps/backend/app/api/international_admissions.py` | 国际升学 REST API |
| 后端 API | `apps/backend/app/api/profile_enhancement.py` | 背景提升 REST API |
| 后端 Schema | `apps/backend/app/schemas/counseling.py` | 升学相关数据模型 |
| 数据库模型 | conftest.py 中引用 | `HongKongMacaoPlanModel`, `InternationalPlanModel`, `ProfileEnhancementPlanModel` |

Navbar 中 "升学" 菜单项指向 `/hongkong-macao`，位于 `Navbar.tsx:96-105`。

### 1.3 删除升学后的遗漏分析

升学模块中的**"背景提升"**概念与需求中的**"能力标签"**有部分语义重叠，但实现方式完全不同：
- 需求中的能力标签是**项目完成后 AI 自动推荐**的，属于成果档案卡的组成部分
- 代码中的背景提升是**独立的 CRUD 计划管理**，与项目和能力标签完全脱钩

**结论：删除升学不会遗漏任何需求功能。** 但需确认"能力标签推荐"功能是否已完整实现（代码中 `capabilityTagsApi.recommend` 已存在）。

### 1.4 附带发现：同样不在需求中的页面

| 页面 | 路由 | 说明 |
|------|------|------|
| `QuestionnaireEngine.tsx` | `/questionnaire-engine` | 问卷引擎，需求中无定义 |
| `KnowledgeSources.tsx` | `/knowledge-sources` | 知识来源管理，需求中无定义 |
| TrackA / TrackE | `/track-a`, `/track-e` | 旧版赛道概念，V3.3 已统一为轻项目/标准研学双模式 |

## 2. 对话菜单 vs AI 工作台：功能重复，架构理解错误

### 2.1 需求定义

V3.3 明确定义了 **AI 对话是 AI 工作台（/create）的核心组件**，不是独立功能：

- 场景化 AI 入口（问问题/解释代码/开始项目/写报告）→ 驱动 AI 对话
- AI 对话面板嵌入工作台三栏布局的中间区域
- 首页的对话是轻量入口，不是完整聊天室
- ZeroClaw 是底座，AI 对话通过 ZeroClaw Gateway API 对接

### 2.2 实际代码

存在 **三个独立的 AI 对话入口**，互不关联：

| 入口 | 文件 | 问题 |
|------|------|------|
| 首页对话 | `Home.tsx:65-114` | 完整聊天界面，占首页主体空间 |
| AI 工作台 | `Create.tsx:60-115` | 场景入口 + 对话，但缺少右侧项目上下文面板 |
| 独立对话页 | `AIAssistantDialogues.tsx` | 完整的会话管理 + 聊天，与工作台功能重复 |

Navbar 中 "对话" 菜单指向 `/assistant-dialogues`，位于 `Navbar.tsx:107-116`。

### 2.3 核心偏差

1. **独立对话菜单不应存在**——AI 对话应嵌入 /create 页面，由场景入口驱动
2. **首页对话不应是完整聊天**——需求中首页是 Hero + 三入口卡片 + 精选 Demo + 灵感墙，对话只是轻量入口
3. **AI 工作台缺少项目上下文面板**——需求定义的三栏布局（场景入口 | AI对话 | 项目上下文）只实现了左两栏
4. **ZeroClaw 集成未落地**——代码中 `useStreamingChat` 直接调后端 chat API，没有通过 ZeroClaw Gateway，也没有上下文头注入（X-Session-Id, X-Project-Id 等）

## 3. 代码编辑器菜单 vs 项目内编辑器：定位错误

### 3.1 需求定义

- 在线代码编辑器是 **AI 工作台的子功能**，路由应为 `/create/code`
- 编辑器应与项目上下文关联，不是独立沙箱
- Monaco + WebContainer(JS) / Pyodide(Python) 是 P1 优先级

### 3.2 实际代码

- 独立的 `/code-studio` 路由，`OnlineCodeStudio.tsx`
- Navbar 中 "代码编辑器" 菜单，位于 `Navbar.tsx:128-138`
- Monaco + iframe/WebContainer 实现完整，但**与任何项目无关**——只是个通用沙箱

### 3.3 核心偏差

1. **代码编辑器不应是顶级菜单**——它应嵌入项目详情页或 AI 工作台
2. **编辑器与项目完全脱钩**——无法加载项目代码、无法保存到项目、无法与 AI 助手联动
3. **需求中的核心流程断裂**——学生完成轻项目 Step2（设计与实现）时，应该能直接在编辑器中写代码并关联到项目

## 4. 审计日志菜单：需求中无用户面

### 4.1 需求定义

V3.3 定义了 `AuditFields` 作为**数据模型层的通用字段**（createdAt/updatedAt/isDeleted 等），是内部治理机制，**不是面向学生的功能**。

### 4.2 实际代码

- 独立的 `/audit-logs` 路由，`AuditLogs.tsx`
- Navbar 中 "审计" 菜单，位于 `Navbar.tsx:139-149`
- 后端 `audit_logs.py` API + `AuditLogModel`

**结论：审计日志应从用户界面移除，保留后端服务即可。**

## 5. 匿名试玩→注册流程：未按需求实现

### 5.1 需求定义（V3.3 §2.4 + MVP附件）

| 场景 | 是否要求注册 | 说明 |
|------|-------------|------|
| 浏览 Demo 墙 | 否 | 允许匿名浏览 |
| 试玩 Demo | 否 | 允许匿名体验 |
| 问一个简单问题 | 否 | 允许匿名试用，次数受限 |
| 点击"我也做一个" | **建议注册** | 创建个人项目前弹出**轻注册** |
| 保存到"我的项目" | 是 | 项目必须归属账号 |
| 生成成果档案卡 | 是 | 需要归属与可追踪 |
| 分享链接 | 是 | 需要稳定链接归属 |

### 5.2 实际代码问题

1. **匿名浏览 Demo 墙**：`/explore/demos` 不需要认证 - 正确
2. **匿名试玩 Demo**：Demo 详情页不需要认证 - 正确
3. **匿名问问题**：首页和 Create 页有匿名聊天限制（5次），用 localStorage 计数 - **可被绕过**（清除 localStorage 即可）
4. **"我也做一个"建议注册**：Fork 流程直接跳到创建项目，而 `/research` 路由被 `ProtectedRoute` 包裹 - **没有"建议注册"的轻提示**
5. **轻注册**：`Register.tsx` 是标准注册页面，不是需求中的"轻注册"（最少信息：邮箱/手机号 + 验证码）
6. **LightRegisterPrompt 组件**：存在但只在聊天次数用完时触发，不在 Fork 流程中触发

### 5.3 关键流程断裂

需求中的核心转化路径：`试玩 Demo → 点击"我也做一个" → 弹出轻注册 → 创建项目`

实际代码中：`试玩 Demo → 点击"我也做一个" → ??? → 需要登录才能进入 /research`

**Fork 按钮的注册引导完全没有实现。**

## 6. 研学9阶段：数据层有，UI层严重偏离

### 6.1 需求定义（V3.3 §3.4.3 + UI布局规格 §5.2）

每个阶段应展示：
- 你现在在哪一步
- 做完这一步会得到什么（成就描述）
- 最小完成标准是什么
- AI 可以怎么帮你

UI 应包含：横向进度条（圆形节点 + 连接线 + 脉冲动画）+ 阶段内容区 + 侧边面板（项目信息 + 证据列表 + AI快捷入口）

### 6.2 实际代码

- `standardStages/` 目录下有 9 个阶段组件 - 数据层完整
- `StandardProjectSteps.tsx` 渲染9阶段 - 逻辑层完整
- 但 UI 严重偏离需求：

| 需求要求 | 实际实现 |
|---------|---------|
| 横向进度条（圆形节点 + 连接线 + 脉冲动画） | Badge 列表，无视觉进度 |
| 中文成就描述（"找到了方向"、"立项成功"） | 英文标签（"Stage 1: Brainstorm"） |
| 侧边面板（项目信息 + 证据 + AI快捷入口） | 无侧边面板 |
| "你在哪一步"/"做完得到什么"/"最小完成标准"/"AI怎么帮你" | 有 gain/minimumDone/aiSupport 但全英文 |
| 阶段内容区动态渲染表单 | 有表单但布局简陋 |

### 7.1 需求定义（UI布局规格 §1.2）

侧边导航只有 **4 项**：探索 | 创造 | 研究 | 互联

Navbar 顶部导航有 **9 项**：首页 | 示例项目 | 创建项目 | 灵感墙 | 升学 | 对话 | 课程库 | 代码编辑器 | 审计

| 需求导航 | 需求路由 | 实际导航 | 实际路由 | 偏差 |
|---------|---------|---------|---------|------|
| 探索 | `/explore` | 示例项目 | `/explore` | 名称不一致 |
| 创造 | `/create` | 创建项目 | `/create` | 名称不一致 |

| - | - | 课程库 | `/course-library` | 应在探索页 Tab 内 |

## 8. 自动化测试：未覆盖 MVP 成功判据

### 8.1 需求定义（MVP附件 §2.2）

4 个核心指标需要追踪：
1. Demo → fork 点击率
2. fork → 轻项目完成率
3. 成果档案卡生成率
4. 成果分享率

### 8.2 实际代码

- 后端 `test_full_journey.py` 测试了 API 层面的完整闭环 - 正确
- 前端 `full-user-journey.spec.ts` 只测试了页面加载，**没有验证核心用户流程**
- **没有任何分析/指标追踪代码**——`useAnalytics.ts` 存在但未集成 MVP 指标
- **没有匿名用户流程的 E2E 测试**
- **没有 Fork 流程的 E2E 测试**

## 9. 偏差汇总与优先级建议

| # | 偏差项 | 严重度 | 影响范围 | 建议优先级 |
|---|--------|--------|---------|-----------|
| 1 | 升学/问卷/知识来源等需求外功能 | 高 | 前端3+页面+后端3+API+DB模型+类型 | P0-删除 |
| 2 | 独立对话菜单（应嵌入创造页） | 高 | 导航+1页面+用户认知混乱 | P0-合并 |

## 10. 根因分析

这些偏差的根因是**开发过程中缺少需求文档的持续对齐**：

1. **升学等功能可能是早期需求的残留**——在 V3.0/V3.1 之前可能存在升学规划的概念，但 V3.3 已完全收口为 STEM 研学，代码没有同步清理
2. **功能拆解时按"技术模块"而非"用户路径"**——对话、编辑器、审计被当作独立模块开发，而非嵌入用户流程
3. **UI 布局规格文档未被前端开发参照**——Sidebar 组件正确实现了4项导航，但 Navbar 又做了一套9项导航，说明规格文档未被严格执行
4. **MVP 闭环未被当作验收标准**——测试只验证了 API 可用性，没有验证用户能否走通完整路径
