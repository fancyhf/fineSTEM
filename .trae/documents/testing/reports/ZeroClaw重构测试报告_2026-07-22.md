## 三、问题清单（给开发 agent）

### 问题 #001

- **用例**: TC-E2E-002 / TC-ZC-AI-002 多轮 stage_00 提问（Playwright 前端渲染）
- **对应 TC-ZC 编号**: TC-ZC-AI-002
- **严重度**: 一般（间歇性，非稳定复现）
- **现象**: 
  - **首次跑（headless）**：第一轮 `ask_question` 卡片正常渲染，点击选项按钮后，AI 回复了文字（"先回答一下这个问题，选完后我们继续第二步——看你有多长时间可用！"），但**第二张 `question-card` 在 90 秒内始终未出现**。失败时页面快照显示：无任何 `data-testid=question-card` 元素，仅有 AI 文字消息。
  - **headed 重跑**：3/3 全部通过，TC-E2E-002 在 6.7s 内完成，第二张卡片正常渲染。
  - Python 脚本直连同一 ZeroClaw daemon 时，3 轮 `ask_question` 全部正确推送（tool_call name=`finestem__ask_question`，tool_result MCP 双层 JSON 解析正确）。
  - 判定为**间歇性问题（flaky）**：后端/ZeroClaw 链路始终正常，前端渲染在特定时序下可能丢失第二轮 `tool_call` 的卡片渲染。
- **复现步骤**（间歇性，不一定每次复现）：
  1. 确保前端 dev server 运行：`cd apps/frontend && npm run dev`
  2. 确保 ZeroClaw daemon 运行：`curl.exe -s http://127.0.0.1:42617/health`
  3. 首次跑（可能失败）：`$env:RUN_AI_E2E="1"; cd apps/frontend/tests; npx playwright test zeroclaw-integration --project=chromium --grep "TC-E2E-002"`
  4. 重跑（通常通过）：同上命令再跑一次
- **期望行为**: 每次点击第一张卡片选项后，AI 应推送第二轮 `ask_question`（时间预算），前端应稳定渲染第二张 `question-card`。
- **实际行为**: 首次跑第二轮卡片未渲染（90s 超时）；headed 重跑正常通过（6.7s）。
- **日志/截图**:
  - 首次跑日志：`logs/2026-07-22/e2e-ai.log`
  - headed 重跑日志：`logs/2026-07-22/e2e-ai-headed.log`
  - 首次跑截图：`apps/frontend/tests/test-results/zeroclaw-integration-ZeroC-b61d6--E2E-002-.../test-failed-1.png`
  - 错误上下文：`apps/frontend/tests/test-results/zeroclaw-integration-ZeroC-b61d6--E2E-002-.../error-context.md`
  - WebSocket 帧数据（证明后端推送正确）：`logs/2026-07-22/ws-multiturn.json`
- **修改建议**: 
  - 后端/ZeroClaw 链路正常（Python 脚本 3 轮全通过），问题在**前端渲染层的时序竞态**。
  - 可能原因：点击选项后前端发送 `[选择]` 格式消息，AI 第二轮回复中的 `tool_call` 帧（`finestem__ask_question`）与第一张卡片的状态清理之间存在竞态——某些时序下 `onQuestions` 回调未被触发或 `questions` 状态被覆盖。
  - 建议在 `useStreamingChat.ts` 的 `tool_call` 分支中加日志，确认多轮 `tool_call` 是否都被接收。考虑 `questions` 状态更新是否用函数式 `setQuestions(prev => [...prev, newQuestion])` 而非覆盖式赋值。
  - 因属间歇性问题且 headed 重跑通过，不阻塞发布，但建议开发 agent 排查并加固。
- **状态**: 待开发确认

### 问题 #002

- **用例**: TC-ZC-AI-003 创建项目（方式一 Python 脚本）
- **对应 TC-ZC 编号**: TC-ZC-AI-003
- **严重度**: 一般

  - 崩溃原因：`UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f60a'`（AI 回复含 emoji 😊，Windows 控制台 GBK 编码无法输出）。
  - 崩溃发生在 `print(f"  回复: {text[:80]}")` 语句（line 154），非 AI/ZeroClaw 问题。
  - dump 文件已完整写入（`ws-multiturn.json`），但 turn 3 的帧数据仅包含 error 帧。
- **复现步骤**:
  1. `cd G:/mediaProjects/fineSTEM/apps/backend`
  2. `python scripts/ws_multi_turn_test.py`
- **期望行为**: 脚本应完成 4 轮对话，第 4 轮 AI 调用 `project_creator` 创建项目。
- **实际行为**: 第 3 轮 print 崩溃，后续流程中断。
- **日志/截图**: `logs/2026-07-22/ws-multiturn.log` + `logs/2026-07-22/ws-multiturn.json`
- **修改建议**: 
  - 在脚本开头加 `sys.stdout.reconfigure(encoding='utf-8')` 或设环境变量 `PYTHONIOENCODING=utf-8`。
  - 或将 print 输出改为写文件，避免 Windows 控制台编码问题。
  - 这是**测试脚本环境问题**，不影响产品代码判定，但阻碍了 TC-ZC-AI-003 的验证。
- **状态**: 待开发确认（属 tests/ 目录，测试 agent 可修）
