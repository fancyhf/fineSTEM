# fineSTEM 测试工作指南 v1.0.0

**版本**: v1.0.0
**日期**: 2026-07-22
**适用对象**: AI Agent（开发 agent / 测试 agent）
**配套文档**: `测试体系总览_v2.0.0.md`（用例编号清单）、`ZeroClaw集成重构_v1.0.0.md`（架构）

> **本指南是测试工作的唯一操作手册。** 任何 AI agent 执行测试相关任务前必须读本文档。

---

## 1. 核心理念：测试是独立工作

测试与开发严格分工，**测试 agent 不许改产品代码**。

```
开发 agent                          测试 agent
    │                                   │
    ├─ 开发功能                          ├─ 编写/维护测试用例（只改 tests/ 目录）
    ├─ 自动跑单元测试（必须通过）         ├─ 执行全场景测试（单元/集成/E2E/Playwright UI）
    ├─ 提交代码                          ├─ 记录日志、测试数据、截图
    │                                   ├─ 产出测试报告 + 问题清单 + 修改建议
    │                                   │
    └─────────────< 拿测试报告检查、修改 >─┘
```

### 1.1 职责边界（红线）

| 行为 | 开发 agent | 测试 agent |
|------|:---:|:---:|
| 修改 `apps/backend/app/`、`apps/frontend/src/` 产品代码 | ✅ | ❌ **禁止** |
| 修改 `apps/backend/tests/`、`apps/frontend/tests/`、`apps/frontend/src/**/*.test.ts` 测试代码 | ✅（自己的测试自己维护）| ✅ |
| 编写新测试用例 | ✅（自己开发的功能）| ✅ |
| 跑单元测试 | ✅ 必须 | ✅ |
| 跑 E2E / Playwright UI 测试 | 可选 | ✅ 必须 |
| 修改测试报告、问题清单 | ❌ | ✅ |
| 修改产品代码修复 bug | ✅ | ❌ **找开发 agent** |

**测试 agent 发现 bug 后的唯一动作**：在测试报告里记录问题 + 复现步骤 + 修改建议，通知开发 agent。**绝不自己动手改产品代码**。

### 1.2 工作流

```
1. 开发 agent 完成功能 + 单元测试 → 提交
2. 测试 agent 拉最新代码 → 跑全套测试
3. 测试 agent 发现问题 → 写进测试报告（不改代码）
4. 开发 agent 读测试报告 → 修复 → 重新提交
5. 测试 agent 回归验证 → 更新报告状态
```

| 层 | 工具 | 目录 | 跑法 | 谁负责 |
|---|---|---|---|---|
| 后端单元 | pytest + pytest-asyncio | `apps/backend/tests/test_*.py` | `pytest` | 开发 agent（功能开发时）+ 测试 agent（回归）|

| 真实对话验证 | Python 脚本 | `apps/backend/scripts/ws_*.py` | `python scripts/ws_multi_turn_test.py` | 测试 agent |

## 3. "一个菜单一组测试"原则

fineSTEM 有 **4 个主菜单 + 2 个辅助页面**，每个菜单对应一组独立测试套件（spec），互不依赖。

| 菜单 | 路由 | 核心功能 | spec 文件命名 |
|------|------|---------|-------------|
| 探索 | `/explore` | Demo 墙、Demo 详情、灵感卡片、成果档案详情 | `explore-*.spec.ts` |
| 创造 | `/create` | AI 对话、PBL 流程、代码编辑器、教学模式 | `create-*.spec.ts` |
| 研究 | `/research` | 项目列表、项目详情、阶段进度、成果生成 | `research-*.spec.ts` |
| 互联 | `/connect` | 社交/分享（如适用）| `connect-*.spec.ts` |
| 账号 | `/login` `/register` `/profile` | 登录注册、用户资料 | `auth-*.spec.ts` |
| 首页 | `/` | 着陆、引导 | `home-*.spec.ts` |

### 命名规范

```
<菜单>-<场景>.spec.ts
```
例：`create-pbl-stage00.spec.ts`、`explore-demo-detail.spec.ts`、`auth-login.spec.ts`

### 每组测试必须覆盖

1. **正常流程**（happy path）：用户完成核心任务的完整路径
2. **异常处理**：网络断开、空数据、非法输入、权限不足
3. **边界值**：超长文本、空列表、并发操作
4. **状态持久化**：刷新页面后状态恢复
5. **跨菜单跳转**：从 A 菜单跳到 B 菜单的链接是否正确

## 4. 标准化测试用例编号

**格式**：`TC-<菜单>-<类型>-<序号>`

| 段 | 含义 | 取值 |
|----|------|------|
| 菜单 | 所属功能域 | `CREATE`/`EXPLORE`/`RESEARCH`/`CONNECT`/`AUTH`/`HOME`/`BE`(后端)/`FE`(前端通用) |
| 类型 | 测试类型 | `UNIT`(单元)/`INT`(集成)/`E2E`(端到端)/`UI`(Playwright UI 实测) |
| 序号 | 3 位数字 | 001 起 |

例：
- `TC-CREATE-E2E-001` = 创造菜单的 E2E 第 1 个用例
- `TC-BE-UNIT-042` = 后端单元第 42 个用例
- `TC-EXPLORE-UI-003` = 探索菜单 Playwright UI 实测第 3 个用例

**编号一旦分配不可复用**。废弃用例标记 `DEPRECATED` 保留编号。

## 6. 各类测试执行方法

### 6.1 后端单元测试

```bash
cd apps/backend

# 全部
python -m pytest -v

# 只跑重构相关
python -m pytest tests/test_stage_constants.py tests/test_tools_gates.py tests/test_check_gate_structural.py tests/test_mcp_server.py -v

# 只跑集成测试
python -m pytest -m integration

# 跑单个用例
python -m pytest tests/test_stage_constants.py::TestCanAdvanceTo::test_skip_multiple_stages_blocked -v
```

**输出**：pytest 自带控制台输出 + `--junitxml=report.xml` 生成结构化报告。

### 6.2 前端单元测试

```bash
cd apps/frontend

# 全部（CI 模式，跑完退出）
npm test

# watch 模式（开发时）
npm run test:watch

# 带覆盖率
npm run test:coverage
```

**输出**：vitest 控制台 + `coverage/` 目录（覆盖率报告）。

### 6.3 Playwright E2E / UI 实测

```bash
cd apps/frontend/tests

# 离线（不含 @ai，默认）
npx playwright test --project=chromium

# 含真实 AI
RUN_AI_E2E=1 npx playwright test --project=chromium

# UI 实测模式（有头浏览器 + 录屏 + 每步截图）
npx playwright test --headed --video=retain-on-failure --screenshot=on

# 只跑某菜单
npx playwright test create-*

# 生成 HTML 报告
npx playwright test && npx playwright show-report ../test-results/e2e-report
```

**输出**：
- 控制台实时输出
- `test-results/` 目录：每个失败用例的截图、视频、error-context.md
- `../test-results/e2e-report/`：HTML 报告（含所有用例状态、耗时、截图）

### 6.4 真实对话验证（ZeroClaw 抓帧）

```bash
cd apps/backend

# 单轮抓帧（确认 tool_call 字段）
python scripts/ws_frame_capture.py

# 多轮 PBL 对话（验证 stage_00 三轮提问 → 创建项目）
python scripts/ws_multi_turn_test.py
```

**输出**：`scripts/ws_frames_dump.json` / `scripts/ws_multiturn_dump.json`（完整帧数据）+ 控制台汇总。

## 9. Playwright UI 实测规范

Playwright 不只是断言通过/失败，还要做**真实用户视角的 UI 实测**——模拟真人点击、观察渲染、截图取证。

### 9.1 必须加 data-testid

产品代码里所有可交互元素**必须**有 `data-testid`（开发 agent 的责任）。测试 agent 只通过 `data-testid` 定位元素，不用 className / xpath（脆弱）。

已加的核心元素：
| 元素 | data-testid |
|------|------------|
| 聊天输入框 | `chat-input` |
| 发送按钮 | `send-button` |
| 用户消息 | `message-user` |
| AI 消息 | `message-assistant` |
| 选项卡片 | `question-card` |
| 继续生成按钮 | `continue-button` |
| 代码编辑器 | `code-editor` |

**开发 agent 新增可交互元素时，必须同步加 data-testid 并在本节登记。**

### 9.2 UI 实测必须包含的取证

每个 UI 实测用例必须产出：
1. **截图**：关键步骤前后各一张（`before`/`after`）
2. **断言**：至少 1 个可见性/内容断言
3. **录屏**（失败时）：Playwright 自动录制

```typescript
// UI 实测示例（必含截图 + 断言）
test('TC-CREATE-UI-001: 发送消息后显示 AI 回复', async ({ page }) => {
  await page.goto(`${CREATE_URL}`);
  await page.getByTestId('chat-input').fill('你好');
  await page.screenshot({ path: 'screenshots/tc-create-ui-001-before.png' });
  await page.getByTestId('send-button').click();
  // 等待 AI 消息出现
  await expect(page.getByTestId('message-assistant').last()).toBeVisible({ timeout: 60000 });
  await page.screenshot({ path: 'screenshots/tc-create-ui-001-after.png' });
});
```

### 9.3 @ai 测试标记规则

需要真实 ZeroClaw daemon + LLM 的测试，在 `describe` 标题里加 `@ai`：
```typescript
test.describe('创造菜单 PBL 流程 @ai', () => { ... })
```
默认不跑（playwright.config 的 `grepInvert: /@ai/`），需 `RUN_AI_E2E=1` 才跑。这样离线测试不受 daemon 可用性影响。

## 10. 测试 agent 标准操作流程（SOP）

每次执行测试任务，按此步骤：

```
1. 拉最新代码
   git pull

2. 记录环境快照
   → 写 logs/<date>/environment.txt

3. 跑后端单元测试
   cd apps/backend && python -m pytest -v --junitxml=../../.trae/documents/testing/logs/<date>/junit-backend.xml
   → 记录结果

4. 跑前端单元测试
   cd apps/frontend && npx vitest run --reporter=json > ../../.trae/documents/testing/logs/<date>/vitest-results.json
   → 记录结果

5. 启动前端 dev server（如 E2E）
   cd apps/frontend && npm run dev &

6. 跑 Playwright 离线测试
   cd apps/frontend/tests && npx playwright test --project=chromium
   → 记录结果 + 收集 test-results/ 截图

7. （如需）启动 ZeroClaw daemon，跑 @ai 测试
   RUN_AI_E2E=1 npx playwright test
   → 记录结果

8. （如需）跑真实对话验证
   cd apps/backend && python scripts/ws_multi_turn_test.py
   → 存 ws_multiturn_dump.json

9. 编写测试报告
   → 按 8.2 模板写 reports/测试报告_<date>_R<轮次>.md
   → 问题清单按严重度排序
   → 给出修改建议（不改代码！）

10. 通知开发 agent
    → "测试报告已出，有 N 个问题，详见 reports/..."
```

---

## 11. 开发 agent 标准操作流程

```
1. 开发功能
   → 在 app/ 或 src/ 写代码

2. 同步加 data-testid（见 9.1）

3. 写单元测试
   → 后端：tests/test_<模块>.py
   → 前端：src/<模块>.test.ts

4. 跑自己的单元测试（必须全通过）
   pytest tests/test_<模块>.py -v
   npx vitest run src/<模块>.test.ts

5. 提交代码

6. 收到测试 agent 的报告后
   → 读 reports/测试报告_<date>.md 的"问题清单"
   → 逐个修复（参考修改建议但不强制）
   → 修复后跑单元测试确认
   → 提交

7. 通知测试 agent 回归
```

## 12. 禁止事项（红线）

### 测试 agent 禁止

- ❌ 修改 `apps/backend/app/` 任何文件
- ❌ 修改 `apps/frontend/src/` 任何**非测试**文件（只能改 `*.test.ts`、`tests/`）
- ❌ 修改 `config.toml`、ZeroClaw 配置
- ❌ 直接 fix bug（只能记录 + 建议修改）
- ❌ 跳过日志记录直接给结论（必须有可追溯的日志/截图）
- ❌ 删除历史测试报告（只能追加新报告）

### 开发 agent 禁止

- ❌ 修改测试报告、问题清单（那是测试 agent 的产出）
