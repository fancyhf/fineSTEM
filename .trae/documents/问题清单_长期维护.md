## 详细问题记录

### Q-001 选项卡丢失（间歇性）

**状态**：🟡 部分修（tool_call 路径已修，但仍有间歇丢失）

**现象**：AI 调用了 `ask_question` 工具（trace 日志可见），但前端没有渲染出选项卡。约每 3-4 次对话出现 1 次。

**根因分析**：
- ZeroClaw 推送 tool_call 帧和 chunk 帧是异步的
- 如果 tool_call 帧在 done 帧之后到达，或 WebSocket 连接不稳定导致丢帧，前端会错过 ask_question
- 前端 tool_call 处理在 `useStreamingChat.ts` line 598-635，依赖 `data.name === 'ask_question'`（经 normalizeToolName）

**修复记录**：
- 2026-07-22：加了 `normalizeToolName` 处理 `finestem__` 前缀
- 2026-07-22：加了 `questionFired` 标志防止 done 帧重复渲染

**仍需修复**：
- 需要加 tool_call 帧重试/缓存机制，确保 done 帧到达时所有 tool_call 已处理
- 或在 done 帧增加延迟检查（done 后等 500ms 再检查是否有遗漏的 ask_question）

**测试要求**：
- 连续 10 轮对话，统计选项卡丢失率（目标 < 5%）
- WS 抓帧：对比 tool_call 帧数量和前端渲染的卡片数量

### Q-002 AI 用文字/表格表达选项，不调 ask_question

**状态**：🟡 部分修（prompt 强化了，但 DeepSeek 模型仍不 100% 遵守）

**现象**：AI 回复里用 markdown 列表（`- 选项A`）或表格（`| 方向 | 做什么 |`）表达选项，而不是调用 `ask_question` 工具。前端不渲染卡片，用户看到文字选项但无法点击。

**根因分析**：
- DeepSeek 模型的工具调用倾向不如 Claude/GPT 强
- AI 在多轮对话后（context 变长）倾向于用文字而非工具
- system_prompt 虽然有规范，但模型不 100% 遵守

**修复记录**：
- 2026-07-22：system_prompt 加 `<CRITICAL>` 标记
- 2026-07-22：SKILL.md 加正反例
- 2026-07-22：done 帧加了 markdown fallback（但这导致了 Q-003）

**仍需修复**：
- prompt 层面已到极限，需要考虑模型升级（如切换到 Claude/GPT）
- 或在后端拦截：检测到 AI 文本含选择意图但无 tool_call 时，后端自动补一个 ask_question

**测试要求**：
- 每个需要提问的 PBL 阶段（stage_00/01/02/04/07）测试 AI 是否调用 ask_question
- 统计 tool_call 使用率（目标 > 90%）

### Q-003 markdown fallback 误把总结/状态汇报解析成选项卡 ⚠️ 当前最严重

**状态**：🔴 未修

**现象**：AI 回复是项目总结或状态汇报（如"当前进度：1.选题完成 2.开题进行中 3.设计待开始"），前端 markdown fallback 把这种编号列表误解析成选项卡，用户看到"请选择"但实际是进度汇报。而 AI 真正想问的问题反而没被渲染。

**根因**：
- `extractQuestionsFromMarkdown` 的过滤不够严格
- `isStatusLine` 只过滤以"当前/阶段/项目"开头的行，但不过滤"1.选题完成 2.开题进行中"这种内容
- Q-002 的 markdown fallback 修复直接导致了这个问题——**fallback 的误伤比它兜底的价值更大**

**修复方向（决策）**：
- **方案 A（推荐）**：完全关闭 markdown fallback。只信任 ask_question tool_call。理由：Q-002 的 prompt 强化 + trace 确认 AI 已在调用工具，fallback 的误识别问题（Q-003）比它解决的兜底问题（Q-002）更严重
- 方案 B：大幅收紧 fallback 过滤——只解析标题**明确含问号或选择动词**的列表，且列表项 ≤4 个且不含句号

**测试要求**：
- 给 AI 发总结性文本（如"总结一下当前进度"），验证不会误产生选项卡
- 给 AI 发真正的选择题文本，验证会正确产生选项卡（仅方案 B）

### Q-004 同一组问题出现重复选项卡

**状态**：🟡 部分修

**现象**：同一张选项卡在屏幕上出现两次。用户点掉一张还有一张。

**根因**：
- 双路径渲染：tool_call 路径渲染一次（onQuestions），done 帧的 markdown fallback 又从 AI 文本解析出同样的选项渲染第二次
- 2026-07-22 已加 `questionFired` 标志修复了 tool_call + fallback 的重复
- 但仍可能有其他重复来源：session 恢复时重复入栈、onQuestions 被调用多次等

**修复记录**：
- 2026-07-22：加 `questionFired` 标志，tool_call 已渲染则 done 帧跳过

**仍需修复**：
- 如果 Q-003 采用方案 A（关闭 fallback），此问题自动消失
- 需验证 session 恢复路径（refresh 页面后 pendingQuestions 恢复）不会重复

**测试要求**：
- 连续 5 轮对话，每轮检查是否有重复卡片
- 刷新页面后检查卡片不重复

### Q-005 AI 反复问已答过的问题

**状态**：🟢 已修（R05 验证通过 2026-07-23）

**现象**：用户已经回答了项目名称，但 AI 在后续轮次再次提问项目名称。或用户已选年级，AI 又问一次年级。R03/R04 测试中 11 轮全部问同一个"年级"问题。

**根因**（两层，R05 最终确认）：
1. **测试代码 bug（主因）**：Playwright 的 `clickFirstOption` 只点了选项按钮，没点"确定"按钮。QuestionCard 设计是"点选项→切换选中，点确定→提交"。只点选项不点确定，`handleQuestionAnswer` 从未被调用，`[选择]` 消息从未发送给 AI。
2. **React 反模式**：`handleQuestionAnswer` 里 `setTimeout(handleSend)` 在 `setPendingQuestions` updater 回调内，React 18 严格模式可能双执行或丢弃副作用。
3. **context 注入不足**：只注入问题标题，没注入具体答案值。

**修复**：
1. 测试代码：`clickFirstOption` 增加 `clickSubmitButton`——点选项后自动点"确定"提交
2. `handleQuestionAnswer`：副作用（`setTimeout(handleSend)`）移出 `setPendingQuestions` updater，在 setState 外面执行
3. `studentProfileRef`：记录 `{title → labels}` 具体答案，注入 `student_profile: 年级=初中; 时间=6小时`
4. system_prompt：加"读完 student_profile 后绝对不要重复问"规则

**R05 验证**（Playwright 有头 TC-DLG-006）：
```
stage_00 轮1: 你现在是哪个年级？
stage_00 轮2: 你这次研学大概有多少时间？
stage_00 轮3: 你对做什么项目有想法了吗？
stage_01 轮1: 你平时课余时间喜欢做什么？（多选）
stage_01 轮2: 你想做哪个游戏项目？
...
```
8 轮全部不同问题，AI 正确推进 ✅

**测试要求**：
- 完成 stage_00（3 轮问答）后，进入 stage_01，验证 AI 不再问年级/时间
- 回答项目名称后，验证 AI 不再问名称
- ⚠️ 测试脚本点击选项后必须点"确定"按钮，否则答案不提交

### Q-006 多选卡片实际无法多选

**状态**：🔴 未修

**现象**：AI 发了 `multiple=true` 的 ask_question，但用户在卡片上只能选一个选项，选第二个时第一个取消选中。

**根因**：
- QuestionCard 组件的 `toggleOption` 逻辑依赖 `isMultiple`（line 55），`isMultiple` 依赖 `data.multiple`
- 需要验证 `data.multiple` 是否从 tool_call 路径正确传递到 QuestionCard
- 可能的断裂点：useStreamingChat 的 `toolCallToQuestion` 或手动构造 QuestionData 时 `multiple` 丢失

**修复方向**：
- 在 QuestionCard 加 debug 日志打印 `data.multiple` 和 `isMultiple`
- 验证 tool_call → QuestionData → showPendingQuestions → pendingQuestions → QuestionCard 全链路

**测试要求**：
- AI 发 `multiple=true` 的 ask_question
- 在卡片上选 2 个选项，验证 2 个都保持选中

### Q-007 AI 只总结不给选项，导致流程无法推进

**状态**：🔴 未修

**现象**：AI 回复了一大段项目总结/分析，但没有调用 ask_question 给出下一步选项，也没有明确告诉用户该做什么。用户不知道如何继续。

**根因**：
- AI 倾向于"分析"而非"提问"（尤其 context 变长后）
- system_prompt 没有强制要求"每个阶段必须有明确的下一步动作"
- markdown fallback 把总结误解析成选项卡（Q-003），让 AI 以为已经给了选项

**修复方向**：
- system_prompt 加规则："每个阶段回复的结尾必须有明确的下一步——要么调用 ask_question，要么明确告诉学生该做什么"
- 关闭 markdown fallback（解决 Q-003 的同时，避免 AI 误以为总结=选项）

**测试要求**：
- 每轮 AI 回复后，验证有明确的下一步动作（ask_question 或文字指引）
- 验证不会出现"只有总结没有动作"的死循环

### Q-008 回复被截断，"继续"按钮无效

**状态**：🟢 已修（2026-07-22）

**现象**：AI 回复中途截断，点击"继续"按钮没有反应。

**根因**：ZeroClaw 的 max_tool_iterations 和 token limit 导致输出截断。

**修复**：max_tool_iterations 从 20 提到 100；加了自动续接机制（_doStreamWithAutoContinue）。

**状态**：🟢 已修（2026-07-22）

**现象**：ZeroClaw 推送 thinking 帧但前端直接 return 丢弃了内容。

**修复**：加了 onThinking 回调，渲染到可折叠的"思考过程"区域。

### Q-010 [选择] 格式识别失败

**状态**：🟢 已修（2026-07-22）

**现象**：学生点击选项卡后前端发送 `[选择] 标题\n回答：选项`，但 AI 不识别这种格式，说"你没选"。

**修复**：system_prompt 加了 [选择] 格式说明；AI 调 skill_state_reader 读项目状态后继续。
