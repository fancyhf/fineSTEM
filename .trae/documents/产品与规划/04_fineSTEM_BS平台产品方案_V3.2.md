## 1. 产品定位

### 1.1 一句话定义

**面向青少年的 STEM 研学助手平台——在线体验、AI 引导、过程驱动，从兴趣到论文一站式。**

### 1.2 与历史定位的演进

| 版本 | 定位 | 核心形态 | AI 角色 |
|------|------|---------|---------|
| v1.0 (白皮书) | "不用背语法，也能让计算机听你的话" | 媒体驱动 + Web终端 + IDE插件 | 苏格拉底导师 |
| v2.0 (PBL体系) | "AI PBL 项目式编程伴学体系" | Web Agent + IDE Agent | 流程引导 + 多角色Agent |
| v3.0 (基础方案) | "青少年 STEM 研学助手平台" | B/S 平台 + ZeroClaw + AI IDE 互联 | ZeroClaw Skill 驱动 |
| **v3.1 (本方案)** | **"青少年 STEM 研学助手平台"** | **同 v3.0 + 转化闭环 + 成果闭环** | **同 v3.0 + 场景化入口 + 渐进式引导** |

### 1.3 核心差异：不是编程平台

```
传统编程平台（豆包编程/Scratch/Code.org）：
  学生 → 学编程 → 做作品 → 展示

fineSTEM v3.1：
  学生 → 发现兴趣 → 立项（工单）→ 研究过程（AI引导）→ 产出（论文/报告/作品）
         ↑                                                    │
         └──────────── AI 助手全程陪伴引导 ←────────────────────┘
```

传统平台教"怎么写代码"，fineSTEM 教"怎么做一个研究项目"——代码只是工具之一。

### 1.4 核心价值主张

1. **过程驱动**：不是只看结果，而是引导完成"选题→开题→开发→结题→论文"全流程
2. **AI 陪伴**：ZeroClaw 驱动的 AI 助手，不是冷冰冰的问答，是懂你的研学导师
3. **成果导向**：最终产出研学报告、小论文、项目作品——可用于综评、升学、竞赛
4. **开放互联**：与 Trae/Cursor/VSCode 等 AI IDE 互通，不是封闭平台
5. **轻量起步**：30 分钟就能完成一个轻项目，不需要一开始就面对完整流程

## 2. 用户模型与路径

### 2.1 核心用户：学生（唯一主要操作者）

学生按参与深度自然分为三级，**不是硬性分组，而是同一平台的不同使用方式**：

```
Level 1 体验者 —— "我来看看"
├── 浏览 Demo 墙，试运行现有项目
├── 融合式学习 Demo（学语文的古诗词卡、学数学的数据可视化）
├── 看代码 + 问 AI 助手"这段代码什么意思"
├── 零门槛，注册即用
└── 典型场景：课间10分钟体验一个物理模拟

Level 2 创作者 —— "我想做一个"
├── 在线选题开题（AI 引导脑爆→开题流程）
├── 简单代码在线编辑运行（沙箱环境）
├── 复杂项目：平台生成项目包，下载到本地用 AI IDE 开发
├── 研学流程全程在线追踪（工单、阶段检查点）
└── 典型场景：花6小时做一个待办清单，产出开题报告+代码+结题报告

Level 3 开发者 —— "我要深度开发"
├── 平台作为索引和参考站（找案例、找资料、找灵感）
├── 下载 fineSTEM Skill 安装到 AI IDE（不限于 Trae）
├── 项目成果可上传回平台展示
├── AI 助手在线咨询（不限于当前项目上下文）
└── 典型场景：用 Trae 做一个 Kaggle 数据分析项目，用到 stem-pbl-guide Skill
```

### 2.2 用户路径与转化目标

三级用户不是静态分类，而是**可逐步升级的路径**。每条路径需要回答四个问题：

#### 体验路径（Level 1）

| 维度 | 设计 |
|------|------|
| **入口** | Demo 墙、课程库、灵感墙 |
| **最小目标** | 试玩一个 Demo + 问 AI 一个问题 |
| **转化触发** | 试玩后点击"我也做一个" |
| **转化机制** | 一键 fork 项目 → 自动创建轻项目 → AI 引导第一步改动 |

#### 创作路径（Level 2）

| 维度 | 设计 |
|------|------|
| **入口** | "我也做一个" fork / 选题推荐 / 场景化 AI 入口"开始项目" |
| **最小目标** | 完成一个轻项目（30分钟~2小时）+ 生成成果档案卡 |
| **转化触发** | 代码变复杂，在线编辑器不够用 / 想做更完整的项目 |
| **转化机制** | AI 提示"你的项目越来越棒了！下载到本地继续开发？" → 生成项目包 + Skill 安装指引 |

#### 深度开发路径（Level 3）

| 维度 | 设计 |
|------|------|
| **入口** | 项目下载中心 / Skill 市场 / 远程 AI 咨询 |
| **最小目标** | 在 AI IDE 中完成一个标准研学项目 |
| **转化触发** | 项目完成，想展示成果 |
| **转化机制** | 上传成果档案卡到灵感墙 → 激励其他学生"我也做一个" |

**核心转化链路**：

```
看见 → 体验 → 理解 → 我也做 → 变成我的 → 完善 → 成果 → 展示 → 激励他人
  │      │      │       │         │         │      │      │
Demo墙  iframe  AI解释  fork项目  轻项目   标准项目  档案卡  灵感墙
```

### 2.3 次要用户

| 角色 | 使用方式 | 权限 | 可见内容 |
|------|---------|------|---------|
| 老师 | 查看学生进度 | 只读 | 项目阶段进度 + 成果档案卡 |
| 家长 | 查看学习报告 | 只读 | 成果档案卡（通过分享链接） |
| 管理员 | 管理内容、Demo、课程 | 全部 | 全部 |

**不需要老师介入教学流程**——AI 助手就是导师。老师和家长通过**成果档案卡分享链接**即可了解学生进展，无需登录系统。

**分享与公开机制**（青少年保护底线）：

| 规则 | 说明 |
|------|------|
| 默认私有 | 成果档案卡生成后默认为私有，仅生成分享链接 |
| 学生主动分享 | 学生可将分享链接主动发给家长/老师 |
| 公开需确认 | 学生主动勾选"申请公开到灵感墙"后才进入审核 |
| 管理员审核 | 管理员审核通过后公开展示 |
| 匿名展示 | 灵感墙公开展示时不显示学生真实姓名和头像，仅用昵称或匿名 |

## 3. 功能模块

### 3.1 模块全景

```
┌──────────────────────────────────────────────────────────────┐
│                     fineSTEM 平台                             │
├────────────┬─────────────────┬───────────────┬───────────────┤
│  探索中心   │   AI 工作台      │  研学流程      │  AI IDE 互联  │
│  (Explore) │   (Create)      │  (Research)   │  (Connect)    │
├────────────┼─────────────────┼───────────────┼───────────────┤
│ Demo 墙    │ 场景化 AI 入口   │ 我的项目       │ Skill 市场    │
│ 课程库     │ AI 对话面板      │ 研学工单       │ 项目导入导出  │
│ 灵感墙     │ 在线代码编辑器   │ 研学文档       │ 远程 AI 咨询  │
│            │ 项目下载中心     │ 成果档案卡     │               │
└────────────┴─────────────────┴───────────────┴───────────────┘
```

### 3.2 探索中心 (Explore)

**定位**：Level 1 体验者的主入口，所有学生的发现层，**核心转化漏斗的起点**。

#### 3.2.1 Demo 墙

**设计原则**：Demo 墙不只是展示，而是**转化入口**。每个 Demo 卡片统一提供 4 个核心动作：

| 动作 | 说明 | 转化目标 |
|------|------|---------|
| **先试玩** | 纯 HTML/JS：iframe 交互嵌入；动态项目：截图+录屏+输入输出示例 | 让学生"看见" |
| **看拆解** | AI 逐步讲解代码 / 项目架构图 + 关键代码片段 | 让学生"理解" |
| **我也做一个** | 一键 fork 项目（不是跳到空白聊天框） | 让学生"动手" |
| **保存到我的项目** | 将 fork 的项目保存到"我的项目" | 让学生"拥有" |

**"我也做一个"fork 机制**：

fork 出来的项目不是原项目的完整复制，而是**最小可改版**：
- 保留核心逻辑
- 去掉复杂功能
- 留出明确的"你可以改这里"标记
- 附带最小任务清单（3个改动建议）
- 预填推荐目标和默认成果页模板

**fork 模板生成策略**（半运营式，不追求全自动）：

| 阶段 | 策略 | 说明 |
|------|------|------|
| MVP | 人工模板 | 为 3 个初始 Demo 各手动制作 1 个高质量 fork 模板 |
| 验证期 | 收集反馈 | 观察学生使用 fork 模板的完成率和改动模式 |
| 增长期 | 半自动生成 | 基于模板规则 + AI 辅助生成，人工审核 |

**每个 fork 模板的固定结构**：

```
fork 模板 = {
  核心代码骨架: 保留原项目最核心的 1-2 个功能,
  删减标记: 注释标注"你可以改这里",
  3 个改动建议: 具体到"把标题改成你喜欢的"这种粒度,
  推荐目标: "完成这3个改动，你就有了自己的版本",
  默认成果页模板: 预填项目名称和来源
}
```

MVP 阶段 3 个 Demo 各做 1 个模板，总共 3 个模板，工作量可控。先验证"fork 机制是否有效"，再验证"fork 是否需要自动化"。

**项目展示方式**：

| 项目类型 | 交互嵌入 | 静态展示增强 | 代码查看 | 下载 |
|---------|---------|-------------|---------|------|
| 纯 HTML/JS | iframe 交互嵌入 | — | 只读代码浏览 | 完整项目包 |
| Flask/Streamlit | — | 截图 + 关键流程录屏 + 输入输出示例 + 项目拆解卡 + 最小可复刻版 | 只读代码浏览 | 完整项目包 |

**初始项目**（已有）：
- 文学知识卡（Flask，端口 4001）→ 静态展示 + 增强内容
- UP主视频分析器（Streamlit，端口 4002）→ 静态展示 + 增强内容
- 智能待办清单（纯 HTML/JS，端口 4003）→ iframe 交互嵌入

**未来扩展**：项目会持续增加，Demo 墙支持动态添加；鼓励制作纯 HTML/JS 版本以支持交互嵌入。

#### 3.2.2 课程库

- **融合式学习 Demo**：跨学科案例（语文+编程的古诗词卡、数学+编程的数据可视化）
- **多维度筛选**：按学科 / 难度 / 年级 / 标签
- **每个课程关联 Demo + AI 问答入口**
- MVP 阶段：框架搭建 + 初始课程元数据

#### 3.2.3 灵感墙

- 展示其他学生的**成果档案卡**（不是项目代码，而是成果展示）
- "我也能做"按钮 → 一键 fork 类似项目
- MVP 阶段：框架搭建，初始展示已有3个项目

### 3.3 AI 工作台 (Create)

**定位**：Level 2 创作者的核心工作区，Level 1/3 的辅助工具。

#### 3.3.1 场景化 AI 入口

**设计原则**：场景化入口优先，自由输入为辅。中学生面对空白聊天框往往不知道怎么开始。

**4 个高频入口**：

| 入口 | 对应场景 | AI 行为 | 上下文注入 |
|------|---------|---------|-----------|
| **问问题** | 看Demo时有疑问 | 基于当前Demo上下文回答 | 自动带入当前 Demo 信息 |
| **解释代码** | 看代码时不懂 | 逐行/逐段解释，配合高亮 | 自动带入当前查看的代码 |
| **开始项目** | 想做一个东西 | 启动 StemPblGuide Skill | 自动带入 fork 来源（如有） |
| **写报告** | 项目做完要总结 | 基于项目过程数据生成 | 自动带入当前项目阶段数据 |

**布局结构**：

```
┌────────────┬──────────────────────┬─────────────────────┐
│  任务入口   │   AI 对话与引导       │  项目上下文面板      │
│            │                      │                     │
│ ○ 问问题   │  [AI 对话区域]        │  当前项目: xxx      │
│ ○ 解释代码 │                      │  阶段: stage_03     │
│ ○ 开始项目 │                      │  工件状态: ✓✓✓○○... │
│ ○ 写报告   │                      │  证据: 3个截图      │
│            │                      │                     │
│ ── 自由 ── │  [输入框]             │  [成果档案卡预览]    │
└────────────┴──────────────────────┴─────────────────────┘
```

#### 3.3.2 在线代码编辑器

- **编辑器**：Monaco Editor（VS Code 同款内核）
- **沙箱运行**：
  - JS：WebContainer（浏览器内运行，零服务器成本）— 优先实现
  - Python：Pyodide（浏览器内运行）— 后续实现
- **AI 辅助**：代码补全 / 代码解释 / 错误调试
- **代码分享**：生成分享链接 / 下载文件
- **限制**：仅支持简单代码运行，复杂项目引导到 AI IDE

#### 3.3.3 项目下载中心

- **生成项目包**：代码框架 + `docs/` + Skill 配置 + `README.md`
- **AI IDE 安装指引**：不限于 Trae，包括 Cursor / VS Code / CodeBuddy 等
- **Skill 安装教程**：如何将 fineSTEM Skill 安装到各种 AI IDE
- **一键下载 .zip**：包含所有启动所需文件

### 3.4 研学流程 (Research)

**定位**：fineSTEM 的核心差异化模块，过程驱动的研学引擎。

#### 3.4.1 我的项目

- **项目列表**：进行中 / 已完成
- **项目详情**：阶段进度条 + 工件状态 + 时间线可视化
- **项目操作**：新建 / 继续 / 导出 / 删除
- **项目模式标识**：轻项目 / 标准研学（见 3.4.3）

#### 3.4.2 双项目模式（核心设计）

**设计原则**：渐进式解锁，不是二选一。轻项目是默认入口，标准研学是升级路径。

##### 轻项目模式

| 维度 | 设计 |
|------|------|
| **适合** | 首次体验或时间有限的学生 |
| **时长** | 30 分钟 ~ 2 小时 |
| **阶段** | 3 步（合并9阶段为核心3步） |
| **目标** | 完成一个可运行 Demo + 一段简短反思 |
| **产出** | 成果档案卡（轻量版） |
| **可升级** | 随时可升级为标准研学模式，已有数据自动继承 |

**轻项目 3 步映射**：

```
轻项目 Step 1: 想法与方向（合并 stage_00~02 初始化+脑爆+开题）
  → 产出：项目名称 + 一句话描述 + 核心功能列表

轻项目 Step 2: 设计与实现（合并 stage_03~07 范围+轨道+设计+计划+开发）
  → 产出：可运行代码 + 关键截图

轻项目 Step 3: 展示与反思（合并 stage_08 验收）
  → 产出：成果档案卡 + 简短反思
```

##### 标准研学模式

| 维度 | 设计 |
|------|------|
| **适合** | 需要完整成果、完整流程和过程沉淀的学生 |
| **时长** | 6 小时 ~ 12 小时+ |
| **阶段** | 完整 9 阶段 |
| **目标** | 完成完整研学项目 |
| **产出** | 成果档案卡 + 研学文档包（开题报告+技术报告+结题报告+论文可选） |

**升级机制**：学生在轻项目任意阶段可以说"我想做得更完整"，系统自动将当前数据映射到标准研学的对应阶段，无需重做。

#### 3.4.3 研学工单（核心差异化）

**设计原则**：引导但不强制。有过程性，但不是必须完成每一步。**交互表达上强调"阶段成就"，弱化"流程负担"**。

每个阶段统一展示 4 个信息：

| 信息 | 说明 | 示例 |
|------|------|------|
| **你现在在哪一步** | 阶段名称和进度 | "Step 2/3: 设计与实现" |
| **做完这一步会得到什么** | 阶段产出 | "你会得到一个可运行的代码" |
| **最小完成标准** | 门禁条件 | "至少1个功能能运行" |
| **AI 可以怎么帮你** | 辅助方式 | "我可以帮你写代码框架" |

**9 阶段流程（标准研学模式，移植自 stem-pbl-guide Skill）**：

| 阶段 | 名称 | 主要产出 | 门禁条件 | 成就描述 |
|------|------|---------|---------|---------|
| stage_00 | 初始化 | 项目目录 + SKILL_STATE.json | 文件存在 | 项目已创建 |
| stage_01 | 脑爆选题 | 脑爆记录 + 选题方向 | 至少1轮；候选≥6；有Top3 | 找到了方向 |
| stage_02 | 开题立项 | 项目立项书 | success_criteria≥2；risks≥2 | 立项成功 |
| stage_03 | 范围裁剪 | 范围约束文档 | must_have≤3；wont_do≥2 | 范围明确 |
| stage_04 | 轨道选择 | 技术轨道规划 | template_id 合法；资源不冲突 | 技术选定 |
| stage_05 | 设计蓝图 | 设计方案 + 代码框架 | 验收用例≥3 | 蓝图完成 |
| stage_06 | 分步计划 | 分步执行计划 | 每步有 run/check/rollback | 计划就绪 |
| stage_07 | 执行开发 | 源代码 + 开发日志 | 至少完成1个milestone | 代码跑起来了 |
| stage_08 | 验收展示 | 验收评估 | 验收项≥2；反思≥2 | 项目完成 |

**灵活入口设计**：
- 学生可以说"我已经有想法了" → AI 帮他快速填完开题卡
- 学生可以说"我不想写论文" → paper_mode=off
- 学生可以说"我只想做个简单 Demo" → 自动进入轻项目模式
- 门禁保证最低质量底线，但路径和深度由学生决定

#### 3.4.4 研学文档

- **文档类型**：开题报告 / 技术报告 / 结题报告 / 论文（可选）
- **AI 辅助生成**：生成框架和引导，不是代写
- **教学模式**：4 种可选（引导式/演示式/动手式/讲解式）
- **导出格式**：PDF / DOCX / PPTX
- **证据素材**：截图 / 图表 / 日志 / 数据（优先自动采集，见 3.4.6）

#### 3.4.5 成果档案卡（核心展示单元）

**定位**：比 PDF 更容易分享、比技术文档更人性化、可积累的成长记录。**一个数据结构，多处复用**——灵感墙展示别人的档案卡，我的项目页展示自己的档案卡，分享给家长/老师的也是档案卡。

**档案卡内容**：

| 字段 | 来源 | 说明 |
|------|------|------|
| 项目名称 | 学生填写 / AI 建议 | — |
| 一句话介绍 | 学生填写 / AI 生成 | 我做了什么 |
| 我解决了什么问题 | 从开题卡提取 | 问题陈述 |
| 我用了什么方法 | 从技术轨道+设计提取 | 技术方案 |
| 项目截图或演示链接 | 自动采集 + 手动补充 | 证据素材 |
| 我的反思 | 学生填写 / AI 引导 | 学到了什么 |
| AI 总结的能力标签 | AI 自动生成 | 如"数据处理"、"界面设计"、"项目管理" |
| 项目模式 | 系统记录 | 轻项目 / 标准研学 |
| 完成时间 | 系统记录 | 实际耗时 |

**能力标签体系**（MVP 最简版）：

项目完成后 AI 自动生成 3 个能力标签，从以下标签库中选择：
- 数据处理 / 界面设计 / 项目管理 / 逻辑推理 / 创意表达 / 代码实现 / 文档撰写 / 问题分析

标签积累在用户档案中，形成**成长记录**。

**分享机制**：
- 生成唯一分享链接
- 家长/老师通过链接查看（无需登录）
- 可选公开到灵感墙
- 默认私有，学生主动勾选后才申请公开
- 管理员审核通过后公开展示
- 灵感墙公开展示时使用昵称或匿名，不显示真实姓名和头像

**下一步推荐**（成果页也是下一项目的起点）：

成果档案卡底部展示"下一步挑战"区域，基于当前项目的能力标签和难度，从题库中匹配 3 个推荐：

```
┌─────────────────────────────────────┐
│  下一步挑战                          │
│                                     │
│  你已经掌握了 [数据处理] [界面设计]   │
│  推荐你试试：                        │
│  → [同难度变体] UP主视频分析器       │
│  → [难度升级] Kaggle 数据建模入门    │
│  → [深度开发] 下载到 Trae 继续完善   │
└─────────────────────────────────────┘
```

MVP 阶段使用规则匹配（能力标签 + 难度 → 题库筛选），不做 AI 推荐算法。

#### 3.4.6 自动证据采集机制

**设计原则**：青少年用户最不擅长"补材料"，系统应自动收集、最后自动编排。

| 采集层级 | 机制 | 示例 |
|---------|------|------|
| **平台自动** | 阶段变更自动记录 | "stage_05→stage_06 于 2026-04-21 14:30" |
| **平台自动** | 代码提交自动留痕 | 每次保存代码自动记录版本 |
| **平台自动** | AI 对话摘要自动生成 | 每次对话结束自动提取关键问答 |
| **半自动** | 学生点"截图"按钮 | 比手动上传简单，一键保存当前画面 |
| **半自动** | AI 提醒保存运行结果 | "要不要保存这个运行结果作为证据？" |
| **手动** | 外部文件上传 | 可选，不强制 |

**自动编排**：报告生成时自动引用已有证据，减少手动补充。

#### 3.4.7 选题推荐

- **多维度启动入口**（不是只有"我来选题"）：

| 启动方式 | 场景 | 实现方式 |
|---------|------|---------|
| 从 Demo 启动 | "我试玩了一个待办清单，想做一个类似的" | Demo 卡片"我也做一个" |
| 从兴趣词启动 | "我喜欢天文和数据图表" | 题库按标签/关键词索引 |
| 从学科任务启动 | "老师让我做一个数学建模展示" | 题库按学科索引 |
| 从比赛题目启动 | "我想参加创新比赛" | 题库按竞赛类型索引 |

- **三轨道题库**（已有）：
  - Web 项目题库（20个，beginner~intermediate）
  - Kaggle/建模题库（10个，beginner~advanced）
  - 硬件创客题库（10个，beginner~advanced）
- **基于兴趣的 AI 推荐**

### 3.5 AI IDE 互联 (Connect)

**定位**：平台与本地 AI IDE 的桥梁，不是竞争关系，是接力关系。

#### 3.5.1 Skill 市场

- **fineSTEM Skill 列表**：stem-pbl-guide 等
- **安装指引**：不限于 Trae，支持 Cursor / VS Code / CodeBuddy 等
- **一键复制安装命令**
- **Skill 使用教程**

#### 3.5.2 项目导入/导出

**三种同步状态**：

| 状态 | 说明 | 适用场景 |
|------|------|---------|
| **导出副本** | 下载 .zip 后默认不持续同步 | 大多数情况 |
| **绑定项目** | 平台与本地建立持续同步关系 | 长期深度开发（后续实现） |
| **冲突处理** | 同一项目两边修改时的优先规则 | 绑定项目场景（后续实现） |

MVP 阶段只实现"导出副本"。绑定同步和冲突处理为后续功能。

- **平台 → 本地**：下载项目包（代码框架 + docs + Skill 配置）
- **本地 → 平台**：上传成果档案卡到灵感墙展示

#### 3.5.3 远程 AI 咨询

- 不在 AI IDE 里也能问 AI
- 项目上下文关联（AI 知道你在做什么项目）
- 代码片段粘贴咨询

## 5. ZeroClaw Skills 规划

### 5.1 核心 Skills

#### StemPblGuide（研学流程引导）

- **来源**：移植自 `.trae/skills/stem-pbl-guide/`
- **功能**：9 阶段状态机 + 研学文档轨 + 4 种教学模式 + 轻项目/标准研学双模式
- **适配**：从 Trae Skill 格式转换为 ZeroClaw Tool Trait
- **关键数据结构**：SKILL_STATE.json 保持兼容，平台和 AI IDE 可互操作
- **新增能力**：轻项目 3 步映射 + 渐进式升级 + 阶段成就描述

#### CodeRunner（在线代码执行）

- **功能**：安全执行学生代码，即时反馈
- **实现**：JS 通过 WebContainer，Python 通过 Pyodide
- **安全**：资源限制 + 超时控制 + 输出过滤

#### KnowledgeBase（知识检索）

- **功能**：fineSTEM 自有内容检索
- **数据源**：课程库 + 案例库 + 题库（topic_library_*.md / track_library.md）
- **检索方式**：向量嵌入 + 关键词搜索
- **多维索引**：支持按学科/兴趣词/竞赛类型等多维度检索

#### DemoExplorer（Demo 体验引导）

- **功能**：解释 Demo 代码 + 引导"我也做一个" + 生成最小可改版 fork
- **场景**：学生看 Demo 时问"这段代码什么意思"、"我想做一个类似的"
- **新增能力**：fork 项目时自动生成最小任务清单和推荐改动点

#### IDEConnector（AI IDE 互联）

- **功能**：生成项目下载包 + Skill 安装指引 + 项目状态同步
- **支持**：Trae / Cursor / VS Code / CodeBuddy 等
- **输出**：.zip 项目包（含代码框架 + docs + Skill 配置 + README）
- **同步语义**：MVP 只支持"导出副本"，绑定同步后续实现

### 5.2 Skill 开发原则

1. **自主开发**：fineSTEM 的 Skill 是核心竞争力，不依赖第三方 Skill 市场
2. **精简聚焦**：只做教育研学领域需要的 Skill，不做大而全
3. **平台/AI IDE 双跑**：同一 Skill 逻辑在 ZeroClaw（平台）和 Trae（本地）都可运行
4. **数据兼容**：SKILL_STATE.json 格式在平台和 AI IDE 之间完全兼容

## 8. 关键设计决策

### 8.1 引导强度：渐进式解锁

- **轻项目是默认入口**，不需要一开始就面对完整9阶段
- **轻项目可随时升级**为标准研学，已有数据自动继承
- 门禁条件保证最低质量底线（如开题卡必须有 success_criteria）
- 学生可以跳过非关键阶段（如"我已经有想法了"可快速填完脑爆）
- AI 会建议但不强制（"建议你再想想其他方向？"而非"你必须完成脑爆"）
- 不需要老师介入，AI 助手就是导师

### 8.2 转化机制：Demo → 项目 → 成果

```
看见 → 体验 → 理解 → 我也做 → 变成我的 → 完善 → 成果 → 展示
  │      │      │       │         │         │      │      │
Demo墙  iframe  AI解释  fork项目  轻项目   标准项目  档案卡  灵感墙
```

- **"我也做一个"不是跳到空白聊天框**，而是 fork 一个最小可改版项目
- fork 项目自带初始代码骨架 + 最小任务清单 + 推荐改动点
- 成果档案卡是核心展示单元，一处生成，多处复用

### 8.3 平台与 AI IDE 的关系：接力而非竞争

```
平台（起跑区）              AI IDE（赛道）
选题 → 开题 → 简单验证  →  深度开发 → 复杂代码
                          ↑
                    项目包下载 + Skill 安装
```

- 平台做轻量体验和流程引导
- AI IDE 做深度开发
- 两者通过项目包和 Skill 无缝衔接
- 不限于 Trae，支持任何 AI IDE

### 8.4 LLM 策略：纯云端

- 不做本地模型部署
- 通过 ZeroClaw Provider Trait 对接云端 LLM
- 支持 Claude / OpenAI / 国内模型等
- 通过 ZeroClaw 统一接口，切换模型无需改业务代码

### 8.5 Demo 展示策略：分层处理

- 纯 HTML/JS 项目：iframe 交互嵌入（最佳体验）
- 动态项目（Flask/Streamlit）：静态展示 + 增强内容（录屏+拆解卡+最小可复刻版）
- 所有项目：支持代码查看 + 项目包下载 + "我也做一个" fork
- 未来鼓励制作纯 HTML/JS 版本以支持交互嵌入

### 8.6 代码沙箱策略：渐进实现

- 第一优先：JS 通过 WebContainer（浏览器内运行）
- 第二优先：Python 通过 Pyodide（浏览器内运行）
- 不做服务器端 Docker 沙箱（成本和安全问题）
- 复杂代码引导到 AI IDE 本地运行

### 8.7 AI 入口策略：场景化优先

- 4 个高频入口（问问题/解释代码/开始项目/写报告）优先于空白聊天框
- 每个入口自动注入上下文（当前 Demo / 当前代码 / 当前项目 / 当前阶段）
- 自由输入作为补充，不作为主入口

### 8.8 证据策略：自动优先

- 平台自动采集 > 半自动提醒 > 手动上传
- 报告生成时自动引用已有证据
- 不依赖学生主动"补材料"

### 8.9 内容运营与审核最小模型

平台是否好用不只取决于 AI 和流程，还取决于持续可用的内容供给。MVP 阶段内容量小，手动维护即可，但需明确职责归属。

| 职责 | MVP 阶段 | 后续 |
|------|---------|------|
| Demo 维护 | 管理员手动添加元数据 | 后台 Demo 管理 CMS |
| fork 模板维护 | 开发者手动制作 3 个模板 | 半自动生成 + 人工审核 |
| 课程元数据 | 管理员手动编辑 JSON | 后台课程管理 |
| 灵感墙审核 | 管理员审核队列 | 审核队列 + AI 辅助预审 |
| 推荐位管理 | 固定排序 | 基于热度/标签/难度推荐 |

**管理员最小权限**：
- Demo 增删改
- fork 模板增删改
- 课程元数据增删改
- 灵感墙审核（通过/拒绝/下架）
- 用户管理（查看/禁用）

## 10. 风险与待决事项

### 10.1 待决事项

| # | 事项 | 影响 | 建议 |
|---|------|------|------|
| 1 | API Server 技术选型 | 开发效率 | 选择与 ZeroClaw 集成最自然的方案 |
| 2 | ZeroClaw Rust SDK 成熟度 | 集成难度 | 先用 JS/Python 绑定验证，Rust SDK 后续跟进 |
| 3 | WebContainer 兼容性 | 代码沙箱 | 验证主流浏览器支持情况 |
| 4 | Pyodide 包支持范围 | Python 沙箱 | 先支持纯 Python 标准库，第三方包逐步添加 |
| 5 | stem-pbl-guide 移植工作量 | 核心功能 | Skill 逻辑复用，交互层需重写为 Web UI |
| 6 | 轻项目→标准研学升级映射 | 数据兼容 | 需定义 3步→9阶段的字段映射规则 |
| 7 | fork 机制的最小可改版生成 | 转化效果 | 需定义"最小可改版"的生成规则和模板 |

### 10.2 风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| ZeroClaw 尚在早期，API 可能变化 | 中 | 高 | 抽象适配层，降低耦合 |
| 云端 LLM 成本 | 高 | 中 | 设置用量限制 + 缓存常见问答 |
| 代码沙箱安全 | 低 | 高 | WebContainer 天然隔离；Pyodide 限制文件系统访问 |
| 学生流失（Level 1 不转化） | 中 | 中 | fork 机制 + 轻项目模式降低起步门槛 |
| 轻项目模式体验过于简单，无法展示差异化 | 中 | 中 | 确保 AI 引导质量 + 成果档案卡有成就感 |
| fork 生成的最小可改版不够好 | 中 | 中 | 先手动制作3个高质量模板，验证后再自动化 |
