# 创造主链路一次性重构解决方案

- version: v1.0.0
- created_at: 2026-06-13 15:10:34.000
- owner: AI Agent
- scope: fineSTEM `创造` 菜单、项目恢复、阶段推进、代码工作台

## 一、背景与问题定义

当前 `创造` 菜单存在四类关键断点：

1. 用户发起新项目后，页面长期停留在前期问答，难以稳定进入编码实现与预览。
2. 历史项目重新打开时，聊天记录、代码内容、当前阶段恢复不完整。
3. 项目阶段推进缺乏单一状态源，前端展示状态与后端真实状态不一致。
4. 报告、验收、论文等末端产出未被纳入统一工作台，导致流程化陪伴学习无法闭环。

这些问题并非单点功能缺失，而是“以对话为中心”的早期实现，与“以项目和流程为中心”的产品目标发生了结构性偏差。

## 二、解决目标

本方案的目标是将 `创造` 重构为“单项目工作台”：

- 一个项目只对应一个真实状态源。
- 一个项目具有可恢复的完整工作区快照。
- 项目流程由阶段状态机驱动，而不是由自由对话驱动。
- 编辑器、预览、证据、报告、论文都成为同一工作台的固定组成部分。

本次实现聚焦三条主线：

1. 统一状态源：`project + skill_state` 全部以 ORM/数据库为准。
2. 完整恢复：新增工作台恢复接口，一次性返回项目、阶段、代码、聊天等内容。
3. 事件驱动：后端输出真实阶段事件，前端只消费真实状态，不再写死阶段。

## 三、总体设计

### 3.1 工作台模型

`创造` 页面改为单项目工作台，但本次优先在现有 UI 上完成关键状态修复，不做整体 UI 推倒重写。

工作台包含以下逻辑区域：

- 项目上下文：项目 ID、项目名称、模式、当前阶段。
- 会话区：用户与 AI 的阶段对话。
- 代码区：当前编辑器语言、代码内容、运行结果。
- 恢复区：历史项目恢复时的一次性快照。

### 3.2 单一状态源

项目相关状态统一由数据库维护：

- `projects`：项目基础信息与当前阶段。
- `skill_states`：阶段状态、历史、阶段产出结构。
- `projects.initial_data`：兼容性工作区字段，承载当前代码、聊天和轻量工作区信息。

运行时不再允许 `skill_state` 走 `memory_db`。

### 3.3 恢复模型

新增工作台恢复接口 `GET /projects/{id}/workspace`，一次性返回：

- `project`
- `progress`
- `workspace.code`
- `workspace.language`
- `workspace.filename`
- `workspace.chat_messages`
- `workspace.preview_html`
- `workspace.saved_at`

前端 `Create.tsx` 和 `ProjectDetail.tsx` 一律走该接口恢复。

### 3.4 阶段驱动

阶段状态更新遵循以下原则：

- 前端不写死 `stage_01` / `stage_07`
- 新建项目、恢复项目、AI 工具创建项目后，当前阶段都以后端返回值为准
- WebSocket 中的 `stage_changed` 事件用于实时刷新前端阶段显示

## 四、详细实施方案

### 4.1 后端改造

#### A. 统一 `runtime_db` 的 `skill_state` 入口

将以下方法改为走 `ProjectRepo`：

- `get_skill_state`
- `create_skill_state`
- `update_skill_state`
- `advance_skill_state`
- `get_project_capability_tags`
- `set_project_capability_tags`

这样项目详情、阶段推进、恢复逻辑全部来自同一份 ORM 数据。

#### B. 统一 `initial_data` 的存取规则

在 `ProjectRepo.update_project()` 中，对 `initial_data` 做统一 JSON 序列化，避免复杂对象直接写入文本列。

新增辅助方法：

- `get_project_workspace()`
- `save_project_workspace()`
- `update_project_stage()`

#### C. 新增工作台恢复接口

新增 `GET /projects/{id}/workspace` 返回结构：

```json
{
  "project": {},
  "progress": {},
  "workspace": {
    "code": "",
    "language": "python",
    "filename": null,
    "chat_messages": [],
    "preview_html": "",
    "saved_at": ""
  }
}
```

#### D. 修正代码与聊天持久化

保留现有 `/code` 与 `/chat` 接口，但都改为统一写入 `initial_data.workspace`，避免散落字段。

#### E. 阶段事件

在 Agent 编排过程中，如果识别到项目阶段变化，输出：

```json
{
  "event": "stage_changed",
  "data": {
    "stage": "stage_03_constraints",
    "stage_name": "范围裁剪"
  }
}
```

同时把阶段写入数据库。

### 4.2 前端改造

#### A. `Create.tsx`

- 恢复逻辑改为优先读取工作台恢复接口返回值。
- 删除项目创建、恢复、工具事件中的阶段写死。
- 在恢复项目时，如果没有代码，则显式回退到默认代码，而不是保留旧状态。
- 代码与聊天自动保存失败时记录错误，不再静默吞掉。

#### B. `ProjectDetail.tsx`

- `进入代码编辑器` 按钮改为只写入最小路由上下文，并调用新工作台恢复接口。
- 不再并行手拼 `getCode + getChatHistory`。

#### C. `api.ts`

- 新增 `projectsApi.getWorkspace(id)`。
- 保留现有 `saveCode/getCode/saveChatHistory/getChatHistory`，供兼容路径使用。

## 六、验收标准

满足以下条件视为本次改造成功：

1. 新建项目后阶段以真实后端状态为准，不再被前端写死。
2. 历史项目可以恢复代码、聊天、当前阶段。
3. 服务重启后，阶段状态不会因 `memory_db` 而回退。
4. `Create` 页面可以响应 `stage_changed` 事件刷新阶段显示。
5. 自动保存失败不再无感，至少在控制台有明确错误信息。
6. 前端构建通过，后端编译通过，关键接口调用成功。

## 七、实施边界

本次是“一次性主链路重构”，但为了控制回归面，先完成“状态、恢复、事件”三条基础链。

不在本次中做的内容：

- `Create` 页完整 UI 视觉重构
- 报告/论文独立表结构拆分
- 多文件工程编辑器与构建系统

这些能力将在主链路稳定后，再基于本次统一状态源继续扩展。
